# Label-Driven UI Library

**Version:** 1.1.1  
**Author:** 김민건 (STE개발팀)  
**Company:** HCT Co.,Ltd.  

---

## 📖 소개 (Introduction)
**Label-Driven UI**는 LabVIEW 환경에서 Class와 Label 정보를 활용하여 UI로 사용되는 VI의 요소(컨트롤 및 인디케이터)를 소프트웨어적으로 원활히 조작할 수 있도록 설계된 개발자용 도구형 라이브러리입니다. 기존의 번거로운 참조 노드 생성 및 연결 작업을 동적 라벨 기반 접근으로 대체하여 아키텍처의 유연성을 극대화합니다.

---

## ✨ 핵심 특징 및 도입 효과

* **개발 속도 및 작업 능률 향상:** UI 요소들을 일괄적으로 조작할 수 있어, 개발자가 UI 코드 작성보다 핵심 비즈니스 로직에 집중할 수 있습니다.
* **코드 재사용성 증가:** 새로운 UI가 추가되더라도 사전 정의된 라벨 명명 규칙만 준수하면 별도의 조건부 코드 추가 없이 즉각적으로 대응 및 확장이 가능합니다.
* **성능 오버헤드 감소:** 클래스 프라이빗 데이터 내부에 유지해야 할 UI 참조 데이터가 대폭 감소하여 메모리 공간을 효과적으로 확보할 수 있습니다.

---

## ⚙️ 시스템 권장 사양 및 도입 조건

### 소프트웨어 요구사항
* **운영체제 (OS):** Windows 10, Windows 11
* **LabVIEW 환경:** 2023 Q3 이상의 LabVIEW 버전

### 아키텍처 필수 요건
* **1:1 대응 구조:** 클래스와 UI 담당 VI가 1:1로 대응해야 합니다. 라이브러리는 각 클래스 정보와 VI 참조 번호를 매핑하여 작동합니다.

### 도입 최적화 환경
* **다수의 동일 유형 UI:** 동일한 유형의 UI 요소가 많은 경우 (예: 10개 이상의 불리언 버튼, 여러 개의 경로 입력란 등) 효율성이 극대화됩니다.
* **체계적인 라벨링:** 라벨 정보가 고유하고 잘 정리되어 있어야 합니다. (예: 모든 불리언 버튼의 라벨을 `~Button`으로 끝나도록 통일)

## 📚 주요 메소드 기능 (API Reference)

주요 기능들은 라이브러리의 `[Method]` 폴더 내에 구현되어 있습니다.

| 메소드 이름 | 기능 설명 |
| :--- | :--- |
| **Set VI Ref** | 라이브러리를 위한 초기 설정. 입력한 클래스의 UI를 담당하는 VI 참조번호를 라이브러리에 설정함 (`This VI` 서버 참조와 함께 사용). 초기화 시 단 한 번 실행. |
| **Get VI Ref** | 라이브러리에 설정한 VI 참조번호를 찾는 기능. |
| **Get Element Ref.vim** | 입력한 라벨을 가진 컨트롤 및 인디케이터의 참조 번호를 반환. |
| **Get All Control Labels** | 모든 컨트롤의 라벨들을 문자열 배열로 반환. |
| **Get All Element Labels** | 모든 컨트롤 및 인디케이터의 라벨들을 문자열 배열로 반환. |
| **Get All Indicator Labels** | 모든 인디케이터의 라벨들을 문자열 배열로 반환. |
| **Set Element Value.vim** | 입력한 라벨을 가진 요소에 값을 입력. 타입이 맞지 않으면 강제 변환 시도(실패 시 오류 반환). |
| **Get Element Value.vim** | 입력한 라벨을 가진 요소에서 값을 취득. 타입 불일치 시 강제 변환 시도(실패 시 오류 반환). |
| **Reinit All Element Values**| VI의 모든 컨트롤 및 인디케이터의 값을 기본값으로 초기화. |
| **Set Element Disable.vim** | 입력한 라벨을 가진 요소의 상태를 `Enabled`, `Disabled`, `Disabled and Grayed out` 중 하나로 설정. |
| **Get Element Disable.vim** | 입력한 라벨을 가진 요소의 현재 입력 상태를 반환. |
| **Set Element Visible.vim** | 입력한 라벨을 가진 요소를 숨기거나 다시 표시. |
| **Get Element Visible.vim** | 입력한 라벨을 가진 요소가 현재 화면에 보이는지 여부를 확인. |

## ⚠️ 부록 및 주의사항 (Troubleshooting)

1. **라벨 오탈자 방지:** 입력한 라벨을 가진 컨트롤/인디케이터가 없을 경우 오류를 반환합니다. 오타 방지를 위해 **정규식 조건 배열 획득**이나 **타입 정의(Type Def) 열거형** 사용을 권장합니다.
2. **다형성 VI 프로브 제한:** `.vim` (다형성/가변형 VI) 확장자로 이루어진 VI는 내부 구조 특성상 Probe 기능을 수행할 수 없습니다.
3. **사용자 인터페이스 텍스트 분리:** 라벨 기반 조작이므로 평소 UI상에 표시되는 텍스트를 임의로 변경할 수 없습니다. UI 표시 텍스트는 **Label 대신 Caption**을 활용하여 분리 적용해야 합니다.
4. **라벨 터미널 입력 타입:** 라벨 입력 터미널에는 문자열, 문자열 배열, 열거형, 열거형 배열을 유연하게 입력할 수 있습니다. (열거형 입력 시 내부적으로 문자열로 자동 변환 처리됨)

## 🔄 패치 노트 (Version History)

### Version 1.2
* **참조 재귀 탐색 최적화:** 탭 컨트롤 등 복잡한 계층 구조의 UI에서 발생하는 참조 획득 문제를 해결하기 위해, VI 속성의 실행(Execution) 옵션을 **공유 복제 할당(Shared Clone Reentrancy)**으로 수정하여 안전한 재귀적 객체 탐색(Recursive Gobjects Traverse)이 가능하도록 구조를 개선했습니다.

### Version 1.1
* **Type Specialization Structure 도입:** 가변형 VI(.vim) 내에 타입 특수화 구조를 적용하여, 단일 조작 및 배열 라벨을 통한 일괄 조작을 하나의 VI에서 통합 지원하도록 고도화했습니다. 이로 인해 열거형 라벨 해석 로직이 간소화되고 직관성이 향상되었습니다.

### Version 1.0
* **초기 아키텍처:** FGV(Functional Global Variable) 기반으로 Class Name - VI Refnum Map을 구성하여 매번 참조를 묶고 푸는 번거로움을 제거했습니다. 초기에는 Variant 기반의 VIM을 활용하여 다양한 타입의 UI 값 반환을 처리했습니다.
